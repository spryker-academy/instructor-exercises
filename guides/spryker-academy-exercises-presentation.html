Spryker Architecture Overview

  ┌─────────────────────────────────────────┐
  │           APPLICATION LAYERS            │
  ├──────────┬──────────┬─────────┬─────────┤
  │   Yves   │   Zed    │  Glue   │ Client  │
  │Storefront│Back Office│REST API│Yves↔Zed │
  ├──────────┴──────────┴─────────┴─────────┤
  │             SHARED LAYER                │
  │     Transfers, Constants, Interfaces    │
  └─────────────────────────────────────────┘
  • Yves — Storefront (customer-facing, stateless, fast)
  • Zed — Back Office (business logic, database access)
  • Client — Bridge for Yves-to-Zed communication
  • Shared — DTOs and interfaces shared across all layers

Zed Module Layers

  ┌────────────────────────────────────┐
  │      Communication Layer           │
  │  Controllers, Plugins, Forms       │
  ├────────────────────────────────────┤
  │        Business Layer              │
  │  Facade ← Factory ← R / W / D     │
  ├────────────────────────────────────┤
  │       Persistence Layer            │
  │  Repository (R) / EntityManager(W) │
  │       ↕ Propel ORM ↕              │
  └────────────────────────────────────┘
  • Communication — HTTP entry point (controllers), no business logic
  • Business — All logic lives here, exposed via Facade (public API)
  • Persistence — Data access. Repository reads, EntityManager writes

Exercise 2: Data Transfer Objects

Spryker's Code Generation System

Goal: Define a MessageTransfer DTO via XML and generate PHP classes

Transfer XML Definition

<!-- src/SprykerAcademy/Shared/HelloWorld/Transfer/helloworld.transfer.xml -->
<transfers>
    <transfer name="Message">
        <property name="idMessage" type="int"/>
        <property name="message" type="string"/>
        <property name="fkCustomer" type="int"/>
        <property name="createdAt" type="string"/>
    </transfer>

    <transfer name="MessageCriteria">
        <property name="idMessage" type="int"/>
        <property name="fkCustomer" type="int"/>
    </transfer>

    <transfer name="MessageCollection">
        <property name="messages" type="Message[]"/>
    </transfer>
</transfers>

What Gets Generated

docker/sdk console transfer:generate

Generates Generated\Shared\Transfer\MessageTransfer with:

  • Typed getters/setters: getMessage(), setMessage(string $message)
  • Strict variants: getMessageOrFail(), setFkCustomerOrFail()
  • Array conversion: toArray(), fromArray(), modifiedToArray()
  • Collection adders: addMessages(MessageTransfer $message)
  • Implements ArrayAccess — works with Symfony forms
Convention: Collection property messages (plural) generates adder addMessages() matching the property name.

Exercise 3: Database Schema

Propel ORM & Schema XML

Goal: Define the pyz_message table with columns, foreign keys, and behaviors

Propel Schema XML

<table name="pyz_message" phpName="PyzMessage">
    <column name="id_message" type="INTEGER"
            primaryKey="true" autoIncrement="true"/>
    <column name="message" type="VARCHAR" size="255"
            required="true"/>
    <column name="fk_customer" type="INTEGER"
            required="false"/>

    <unique name="pyz_message-message">
        <unique-column name="message"/>
        <unique-column name="fk_customer"/>
    </unique>

    <foreign-key foreignTable="spy_customer"
                 onDelete="CASCADE">
        <reference local="fk_customer"
                   foreign="id_customer"/>
    </foreign-key>

    <behavior name="timestampable"/>
</table>

Key Propel Concepts

docker/sdk console propel:install
# = propel:schema:copy + propel:model:build + propel:diff + propel:migrate
  • phpName — PHP class name: PyzMessage, PyzMessageQuery
  • Magic methods — findOneByIdMessage(), filterByFkCustomer()
  • Timestampable behavior — Auto-manages created_at / updated_at
  • Foreign keys — onDelete="CASCADE" = delete messages when customer is deleted
  • Composite unique — Same customer can't post duplicate text, different customers can

Exercise 4: Module Layers - Back Office

Persistence, Business, Communication

Goal: Create, read, and delete messages through the three-layer architecture

Persistence Layer: Repository (Read)

class HelloWorldRepository extends AbstractRepository
{
    public function findMessage(MessageCriteriaTransfer $criteria): ?MessageTransfer
    {
        $query = $this->getFactory()->createMessageQuery();

        if ($criteria->getIdMessage()) {
            $entity = $query->findOneByIdMessage($criteria->getIdMessage());
        }

        return $entity
            ? $this->getFactory()->createMessageMapper()
                ->mapEntityToMessageTransfer($entity)
            : null;
    }
}
Rule: Repository = read-only. Never call save() or delete() here.

Persistence Layer: EntityManager (Write)

class HelloWorldEntityManager extends AbstractEntityManager
{
    public function createMessage(MessageTransfer $transfer): MessageTransfer
    {
        $entity = new PyzMessage();
        $entity->fromArray($transfer->modifiedToArray());
        $entity->save();

        return $this->getFactory()->createMessageMapper()
            ->mapEntityToMessageTransfer($entity);
    }

    public function deleteMessage(int $idMessage): bool
    {
        $entity = $this->getFactory()->createMessageQuery()
            ->findOneByIdMessage($idMessage);

        if (!$entity) { return false; }

        $entity->delete();
        return true;
    }
}

Business Layer: SOLID Separation

ClassDependencyMethodsResponsibility
MessageReaderRepositoryInterfacefindMessage(), findMessagesByCustomer()Read only
MessageWriterEntityManagerInterfacecreate()Write only
MessageDeleterEntityManagerInterfacedelete()Delete only
// Each business model has ONE constructor dependency and ONE responsibility
class MessageDeleter
{
    public function __construct(
        protected HelloWorldEntityManagerInterface $entityManager,
    ) {}

    public function delete(int $idMessage): bool
    {
        return $this->entityManager->deleteMessage($idMessage);
    }
}
Single Responsibility: Writer does NOT have delete(). Deleter does NOT have create(). Each class has one reason to change. The Facade orchestrates all three via the Factory.

Business Layer: Facade & Factory

// Facade = Module's public API (one method per use case)
class HelloWorldFacade extends AbstractFacade
{
    public function createMessage(MessageTransfer $t): MessageTransfer {
        return $this->getFactory()->createMessageWriter()->create($t);
    }
    public function findMessagesByCustomer(MessageCriteriaTransfer $c): MessageCollectionTransfer {
        return $this->getFactory()->createMessageReader()->findMessagesByCustomer($c);
    }
    public function deleteMessage(int $id): bool {
        return $this->getFactory()->createMessageDeleter()->delete($id);
    }
}
// Factory = Wiring (each create* instantiates with the right dependency)
class HelloWorldBusinessFactory extends AbstractBusinessFactory
{
    public function createMessageReader(): MessageReader {
        return new MessageReader($this->getRepository());
    }
    public function createMessageWriter(): MessageWriter {
        return new MessageWriter($this->getEntityManager());
    }
    public function createMessageDeleter(): MessageDeleter {
        return new MessageDeleter($this->getEntityManager());
    }
}

Mapper Pattern

class MessageMapper
{
    public function mapEntityToMessageTransfer(
        PyzMessage $entity,
        MessageTransfer $transfer = new MessageTransfer(),
    ): MessageTransfer {
        return $transfer->fromArray($entity->toArray(), true);
    }
}
  • Converts between ORM entities and Transfer objects
  • Lives in the Persistence layer (next to Repository/EntityManager)
  • Default parameter = fresh transfer if none provided

The Stub: Calling Zed from Yves

class HelloWorldStub
{
    public function __construct(
        protected ZedRequestClientInterface $zedRequestClient,
    ) {}

    public function findMessage(MessageCriteriaTransfer $criteria): MessageResponseTransfer
    {
        /** @var MessageResponseTransfer $response */
        $response = $this->zedRequestClient->call(
            '/hello-world/gateway/find-message',
            $criteria,
        );
        return $response;
    }
}
Gateway path convention:
/module-name/gateway/action-name
deleteMessageAction() → /hello-world/gateway/delete-message

GatewayController in Zed

class GatewayController extends AbstractGatewayController
{
    public function findMessageAction(
        MessageCriteriaTransfer $criteria
    ): MessageResponseTransfer {
        return $this->getFacade()->findMessage($criteria);
    }

    public function deleteMessageAction(
        MessageCriteriaTransfer $criteria
    ): MessageResponseTransfer {
        $response = new MessageResponseTransfer();
        $deleted = $this->getFacade()
            ->deleteMessage($criteria->getIdMessage());
        $response->setIsSuccessful($deleted);
        return $response;
    }
}
  • Extends AbstractGatewayController (not AbstractController)
  • Transfer parameter = auto-deserialized from request
  • Return transfer = auto-serialized in response

DependencyProvider Pattern

// Every external dependency goes through the DependencyProvider
class HelloWorldPageDependencyProvider extends AbstractBundleDependencyProvider
{
    public const string CLIENT_HELLO_WORLD = 'CLIENT_HELLO_WORLD';

    public function provideDependencies(Container $container): Container
    {
        $container->set(
            static::CLIENT_HELLO_WORLD,
            fn () => $container->getLocator()->helloWorld()->client(),
        );
        return $container;
    }
}

// Factory retrieves it
class HelloWorldPageFactory extends AbstractFactory
{
    public function getHelloWorldClient(): HelloWorldClientInterface
    {
        return $this->getProvidedDependency(
            HelloWorldPageDependencyProvider::CLIENT_HELLO_WORLD,
        );
    }
}

Configuration Stack

// 1. Constants interface (Shared layer)

$config[HelloWorldConstants::MY_CONFIG_VALUE] = 'Hello from config!';

Exercise 7: Extending Core Modules

CustomerPage, Forms, Delete, AJAX

Goal: Add customer messages to the Yves profile area by extending Spryker's CustomerPage module

Extending Core Modules: The Pattern

Extend the core class in your namespace, override what you need:

// Extend the core DependencyProvider
class CustomerPageDependencyProvider
    extends SprykerCustomerPageDependencyProvider
{
    public const string CLIENT_HELLO_WORLD = 'CLIENT_HELLO_WORLD';

    public function provideDependencies(Container $container): Container
    {
        $container = parent::provideDependencies($container); // Keep core deps
        $container = $this->addHelloWorldClient($container);  // Add ours
        return $container;
    }
}

// Extend the core Factory
class CustomerPageFactory extends SprykerCustomerPageFactory
{
    public function getHelloWorldClient(): HelloWorldClientInterface { ... }
    public function createMessageForm(MessageTransfer $t): FormInterface { ... }
}

// Extend the core RouteProviderPlugin
class CustomerPageRouteProviderPlugin
    extends SprykerCustomerPageRouteProviderPlugin
{
    public function addRoutes(RouteCollection $c): RouteCollection
    {
        $c = parent::addRoutes($c); // Keep all core routes
        $c = $this->addCustomerMessagesRoute($c); // Add ours
        return $c;
    }
}

Symfony Forms in Spryker Yves

// Form class — defines fields and validation only (no SubmitType)
class MessageForm extends AbstractType
{
    public const string FIELD_MESSAGE = 'message';

    public function buildForm(FormBuilderInterface $builder, array $options): void
    {
        $builder->add(static::FIELD_MESSAGE, TextType::class, [
            'label' => 'Your message',
            'constraints' => [new NotBlank()],
        ]);
    }
}
// Usage in controller — data-mapped to a Transfer object
$form = $this->getFactory()->createMessageForm(new MessageTransfer());
$form->handleRequest($request);

if ($form->isSubmitted() && $form->isValid()) {
    $messageTransfer = $form->getData();   // Already a MessageTransfer!
    $messageTransfer->getMessage();         // Contains the form input
}
  • Data binding: Pass MessageTransfer as form data — getData() returns the hydrated transfer
  • ArrayAccess: Transfers implement it, so Symfony maps fields to properties automatically
  • No SubmitType: Button rendered in Twig for Spryker CSS control

Form Factory Chain in Spryker

  CustomerPageFactory
      │
      ├─ createCustomerFormFactory()
      │     └─ FormFactory (extends AbstractFactory)
      │          └─ getFormFactory()
      │               └─ Symfony FormFactory
      │                    └─ create(MessageForm::class, $transfer)
      │
      └─ Returns: FormInterface (CSRF + validation + data binding)
Spryker doesn't expose Symfony's FormFactory directly on module factories. Use the proxy chain: createCustomerFormFactory()->getFormFactory()->create()

Rendering Forms in Twig

{# Render the form with Symfony helpers #}
{{ form_start(data.messageForm) }}
    <div class="form spacing-bottom">
        {{ form_widget(data.messageForm.message) }}
    </div>
    <button type="submit" class="button button--success">
        Add Message
    </button>
{{ form_end(data.messageForm) }}
{# form_end renders remaining fields including the CSRF _token #}
  • form_start — Opens <form> tag with action and method
  • form_widget — Renders the input field (applies Spryker's form theme)
  • form_end — Renders hidden fields (_token) and closes <form>
  • Never use render_rest: false on form_end — it skips the CSRF token!

Controller Best Practices

class MessageController extends AbstractCustomerController
{
    public function listAction(Request $request): View|RedirectResponse
    {
        $customer = $this->getLoggedInCustomerTransfer();

        // Strict validation — fails fast if customer has no ID
        $messageCriteria = new MessageCriteriaTransfer();
        $messageCriteria->setFkCustomerOrFail($customer?->getIdCustomer());

        // Data-mapped form — no manual field extraction
        $form = $this->getFactory()->createMessageForm(new MessageTransfer());
        $form->handleRequest($request);

        if ($form->isSubmitted() && $form->isValid()) {
            $transfer = $form->getData(); // Already a MessageTransfer!
            $transfer->setFkCustomer($customer->getIdCustomer());
            $this->getFactory()->getHelloWorldClient()->createMessage($transfer);
            return $this->redirectResponseInternal(static::ROUTE_CUSTOMER_MESSAGES);
        }
        // ...
    }

    public function deleteAction(Request $request): RedirectResponse
    {
        // Safe ID casting — throws on non-numeric/zero
        $idMessage = $this->castId($request->request->get('idMessage'));
        // ...
    }
}

Template Override Mechanism

Spryker resolves Twig templates in priority order:

  1. src/Pyz/Yves/CustomerPage/Theme/... (project level — highest priority)
  2. src/SprykerAcademy/Yves/CustomerPage/Theme/...
  3. vendor/spryker-shop/customer-page/.../Theme/... (core — lowest priority)

To override a core template, create a file at the same relative path in your project namespace.

Bonus: AJAX in Spryker Yves

The Component Trio Pattern

No jQuery. No custom JS. Pure Spryker components from ShopUi.

The Four AJAX Components

ComponentRoleEvents
ajax-provider XMLHttpRequest wrapper Dispatches fetching / fetched
ajax-form-submitter Intercepts form submit via data-* trigger Listens for click / change
ajax-renderer Parses JSON response, updates DOM Listens for fetched
ajax-loader Shows/hides spinner Listens for fetching / fetched

AJAX Flow

  User clicks [Add Message]
      │ (data-message-ajax-submit + formaction)
      ▼
  AjaxFormSubmitter → preventDefault, collect FormData
      ▼
  AjaxProvider.fetch(formData)
      │ dispatches "fetching" → Loader shows spinner
      │ POST /customer/messages/async/add
      ▼
  MessageAsyncController::addAction()
      │ Creates message, re-renders table
      │ Returns JSON: { messages, content }
      ▼
  AjaxProvider → dispatches "fetched" → Loader hides
      ▼
  AjaxRenderer.render()
      │ content → replaces target innerHTML
      │ messages → UPDATE_DYNAMIC_MESSAGES event
      │ mount() → re-init JS in new HTML
      ▼
  Table updated. No page reload!

Async Controller Response

class MessageAsyncController extends AbstractCustomerController
{
    public function addAction(Request $request): JsonResponse
    {
        // ... validate form, create message ...

        return $this->jsonResponse([
            'messages' => $this->renderView(
                '@ShopUi/.../flash-message-list.twig'
            )->getContent(),
            'content' => $this->getTwig()->render(
                '@CustomerPage/views/message/message-async.twig',
                ['messages' => $collection, 'messageForm' => $form],
            ),
        ]);
    }
}
JSON contract:
messages → rendered flash message HTML (success/error)
content → rendered HTML fragment to replace in the DOM

Key Takeaways

  • Layered Architecture — Communication → Business → Persistence
  • Facade Pattern — Module's public API, single entry point
  • Factory Pattern — All instantiation goes through factories
  • DependencyProvider — External dependencies injected via container
  • Transfer Objects — Generated DTOs for type-safe data passing
  • Client/Stub — Yves talks to Zed through BackendGateway
  • SOLID — Reader/Writer/Deleter separation, single responsibility
  • Module Extension — Extend core classes at project level, call parent::
  • Forms — Data-mapped to transfers, form factory chain, CSRF built-in
  • AJAX — Component trio: Provider + FormSubmitter + Renderer
